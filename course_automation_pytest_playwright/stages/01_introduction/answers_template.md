# Ответы на теоретические вопросы

## 1. Что такое unit тест и зачем он нужен?

**Ответ:**

Unit тест (модульный тест) - это тест, который проверяет работу отдельной функции, метода или класса в изоляции от других компонентов системы.

**Зачем нужен:**
- Проверяет правильность работы отдельных компонентов
- Помогает находить баги на ранних этапах разработки
- Дешевле в поддержке по сравнению с другими типами тестов
- Быстро выполняется
- Служит документацией к коду

---

## 2. В чём разница между integration и E2E тестами?

**Ответ:**

**Integration тесты:**
- Проверяют взаимодействие между компонентами системы
- Тестируют интерфейсы и API
- Меньше охват, чем E2E, но более стабильные
- Пример: тестирование взаимодействия frontend с backend API

**E2E (End-to-End) тесты:**
- Проверяют полные пользовательские сценарии
- Симулируют реальное поведение пользователя
- Охватывают всю систему целиком
- Самые дорогие в поддержке
- Пример: тестирование полного процесса покупки товара

---

## 3. Какие преимущества даёт автоматизация тестирования?

**Ответ:**

**Экономические преимущества:**
- Снижение затрат на ручное тестирование
- Быстрое выполнение регрессионных тестов
- Возможность параллельного запуска тестов

**Качественные преимущества:**
- Повышение точности (исключение человеческого фактора)
- Лучшее покрытие тестами
- Раннее обнаружение дефектов
- Повышение уверенности в качестве продукта

**Организационные преимущества:**
- Стандартизация процессов тестирования
- Улучшенная документация
- Возможность ночных сборок и CI/CD

---

## 4. Когда автоматизация может быть неэффективной?

**Ответ:**

**Ситуации, когда автоматизация неэффективна:**
- Одноразовые тесты, которые не будут повторяться
- Ад-хок и исследовательское тестирование
- Когда стоимость создания автоматизированных тестов выше потенциальной экономии
- При частых изменениях в требованиях
- Для тестирования UX/UI, где важна человеческая оценка
- Когда тестирование требует творческого подхода

**Правило:** Автоматизируйте только стабильные, повторяющиеся сценарии с высоким ROI.

## 5. Почему мы выбрали именно pytest и Playwright?

**Ответ:**

**Выбор pytest:**
- Простой и интуитивный синтаксис
- Мощная система фикстур для управления тестовым окружением
- Богатая экосистема плагинов
- Отличная интеграция с другими инструментами
- Широкое сообщество и хорошая документация
- Поддержка асинхронного кода

**Выбор Playwright:**
- Современная архитектура с автоматическим ожиданием
- Поддержка всех основных браузеров (Chromium, Firefox, WebKit)
- Мощные селекторы и локаторы
- Встроенная поддержка mobile эмуляции
- Хорошая производительность
- Активная разработка и поддержка

**Сравнение с альтернативами:**
- Selenium: более старый, медленнее, сложнее в настройке
- Cypress: только для JavaScript, ограниченная поддержка браузеров
- Robot Framework: менее гибкий, сложнее для программистов
